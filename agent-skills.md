# Clawbooks Agent Skills

You are the accountant. Clawbooks is your ledger, validation surface, and reporting toolkit.

The ledger (`ledger.jsonl`) is an append-only JSONL file. Every row is a financial event. Never edit it directly — use `clawbooks record` or `clawbooks batch` for all writes.

The policy (`policy.md`) defines entity-specific accounting rules: basis, recognition, categories, thresholds. Read it before classifying anything.

## CLI Reference

All commands: `clawbooks [--books DIR] <command> [period] [flags]`

Period formats: `2026` (year), `2026-03` (month), `2026-01/2026-06-30` (range).

### Import

| Command | Purpose |
|---------|---------|
| `import scaffold <kind>` | Create mapper templates. Kinds: `statement-csv`, `generic-csv`, `fills-csv`, `manual-batch`. Use `--out DIR` to set output path. |
| `import check <events.jsonl>` | Validate staged events before append. Use `--statement profile.json` for balance/total checks, `--save-session` to record validation, `--mappings PATH` for vendor mappings. |
| `import reconcile <events.jsonl> --statement profile.json` | Compare staged events to statement expectations (count, debits, credits, balances). |
| `import sessions list` | List saved import validation sessions. |
| `import mappings suggest` | Surface recurring classification patterns from ledger history. |

### Write

| Command | Purpose |
|---------|---------|
| `record '<json>'` | Append one event to ledger. |
| `batch` | Append JSONL events from stdin. Enforces sign conventions, generates IDs, hash-chains. |

### Inspect

| Command | Purpose |
|---------|---------|
| `log [--last N]` | Print recent events (default 20). |
| `stats` | Ledger overview: event count, sources, types, date range. |
| `context [period]` | Policy-aware context envelope with events for reasoning. Use `--include-policy` to inline policy. |

### Verify & Review

| Command | Purpose |
|---------|---------|
| `verify [period]` | Integrity checks: duplicates, sign consistency, balance validation. Use `--balance N`, `--opening-balance N`, `--currency C`. |
| `reconcile [period] --source S` | Compare imported totals to expectations. Flags: `--count N`, `--debits N`, `--credits N`, `--opening-balance N`, `--closing-balance N`, `--date-basis ledger\|transaction\|posting`, `--gaps`. |
| `review [period]` | Materiality-first queue of items needing classification review. Filter: `--confidence inferred,unclear`, `--min-magnitude N`, `--group-by category\|source\|type`. |
| `review batch <period> --out FILE --action confirm\|reclassify` | Generate bulk review actions. |

### Report

| Command | Purpose |
|---------|---------|
| `summary [period]` | Aggregates by type, category, month, source, currency. Cash flow, movement summary, report sections, settlement views, review materiality. This is your P&L. |
| `assets` | Asset register with depreciation. Filter: `--category C`, `--life N`, `--as-of DATE`. |
| `documents [period]` | Settlement and aging views: receivables, payables, unmatched cash. |
| `pack [period] --out DIR` | Audit pack: CSVs + JSON + policy snapshot. |

### Maintenance

| Command | Purpose |
|---------|---------|
| `snapshot [period] [--save]` | Compute or save a balance checkpoint. |
| `compact <period>` | Archive old events, save snapshot, shrink ledger. |
| `policy [lint]` | Print or lint current policy. |

## Event Schema

### Envelope

Every ledger event:

```json
{
  "ts": "2026-03-18T09:30:00.000Z",
  "source": "statement_import",
  "type": "expense",
  "data": {
    "amount": -42.50,
    "currency": "USD",
    "description": "Office supplies",
    "category": "office",
    "confidence": "clear",
    "transaction_date": "2026-03-17",
    "source_doc": "bank-march-2026.csv",
    "source_row": "3",
    "source_hash": "abc123"
  }
}
```

`id` and `prev` are generated by the CLI on write. Do not set them manually.

### Types and Sign Conventions

**Stored negative** (outflows): `expense`, `tax_payment`, `owner_draw`, `fee`, `dividend`, `loan_repayment`, `refund`, `transfer_out`, `withdrawal`

**Stored positive** (inflows): `income`, `deposit`, `equity_injection`, `loan_received`, `transfer_in`, `refund_received`, `grant`

**Documents** (sign from `data.direction`): `invoice`, `bill` — use `issued` (positive) or `received` (negative)

**Meta** (no sign enforcement): `snapshot`, `reclassify`, `correction`, `confirm`, `opening_balance`

**Assets**: `disposal`, `write_off`, `impairment`

Set `amount` to the transaction magnitude. The CLI enforces the correct sign based on type.

### Key Data Fields

| Field | When to use |
|-------|-------------|
| `amount`, `currency` | Always (except meta types). |
| `description` | Always. Human-readable transaction label. |
| `category` | Always for cash events. Classify per policy.md. |
| `confidence` | Always. `clear` = unambiguous, `inferred` = reasonable guess, `unclear` = can't classify. |
| `transaction_date` | Statement imports. The date the transaction occurred. |
| `posting_date` | Statement imports. The date the bank posted it. |
| `source_doc` | Always for imports. The filename of the source document. |
| `source_row` | Always for row-based imports. Row index or label from source. |
| `source_hash` | Always for imports. SHA-256 fingerprint of the source file. |
| `ref` | Stable identity from source (check number, transaction ID). |
| `counterparty` | When identifiable from source. |
| `recorded_via` | Set by the system. Do not override. |

## Import Workflows

### CSV / TSV / JSONL (Statement Import)

1. Read the source file to understand its shape (columns, date format, amount format, headers).
2. `clawbooks import scaffold statement-csv --out generated/imports/statement-csv` to get mapper templates.
3. Edit `mapper.mjs` (or `mapper.py`) to parse the source file into canonical events. The mapper reads the source and writes JSONL to stdout.
4. Run the mapper: `node generated/imports/statement-csv/mapper.mjs uploads/source.csv > generated/staged.jsonl`
5. `clawbooks import check generated/staged.jsonl` — validate before append. Add `--statement generated/imports/statement-csv/statement-profile.json` if you have balance expectations.
6. `clawbooks batch < generated/staged.jsonl` — append to ledger.
7. `clawbooks verify` — check integrity.
8. `clawbooks review` — check classifications.

If import check fails, fix the mapper and re-run. Do not skip validation.

### PDF / Image Sources

There is no scaffold for PDFs. Extract events directly by reading the document:

1. Read the PDF/image to extract transaction data.
2. Build canonical events as JSONL (one per transaction), writing to `generated/staged.jsonl`.
3. Every event must include provenance: `source_doc`, `source_hash`, and `source_row` (page number or logical row).
4. `clawbooks import check generated/staged.jsonl` — validate.
5. `clawbooks batch < generated/staged.jsonl` — append.
6. `clawbooks verify` and `clawbooks review`.

### Direct Batch (Simple Cases)

For small or simple imports where a mapper isn't needed:

1. Build JSONL events directly and write to `generated/staged.jsonl`.
2. `clawbooks import check generated/staged.jsonl`
3. `clawbooks batch < generated/staged.jsonl`
4. Verify and review.

## Reporting

### P&L / Financial Summary

`clawbooks summary` gives you everything for a P&L: aggregates by type, category, month; cash flow (inflows, outflows, net); movement summaries. Filter with a period or `--source` if needed.

Do **not** hardcode a source filter unless the user specifically asks about one source. `clawbooks summary` without a source filter gives the full picture.

### Verification

Always run `clawbooks verify` after any import. If you know the expected closing balance, use `--balance N`. Use `--opening-balance N` if you have the statement opening balance.

### Review Queue

`clawbooks review` shows items sorted by materiality that need human or agent review. Items with `confidence: inferred` or `unclear` appear here. After reviewing, generate `confirm` or `reclassify` events via `clawbooks review batch`.

### Assets

`clawbooks assets` shows the fixed asset register. Events with `data.capitalize: true` enter the register. Track lifecycle via `disposal`, `write_off`, `impairment` events.

### Documents

`clawbooks documents` shows receivables/payables, settlement status, and aging. Match cash events to invoices/bills via `data.invoice_id`.

## Operations Module (Programmatic)

When available as a Node.js dependency, clawbooks exports these from `clawbooks/operations`:

- `buildStats(events)` — ledger overview
- `buildSummary({ all, source?, period? })` — full reporting aggregates
- `buildContext({ all, ... })` — policy-aware context
- `analyzeVerification(events, opts?)` — integrity checks
- `buildReconciliation({ all, source, ... })` — compare to expectations
- `buildReviewQueue({ all, source?, ... })` — review items
- `buildReviewBatch({ all, ... })` — generate review actions
- `buildAssetReport({ all, ... })` — asset register
- `buildDocumentReport({ all, ... })` — settlement views
- `prepareBatch({ existingLines, input })` — sign enforcement, dedup, hash-chaining
- `buildImportCheck({ ... })` — validate staged events

These are useful for fast programmatic access without spawning CLI processes.

## Key Rules

1. **Never edit `ledger.jsonl` directly.** All writes go through `clawbooks record` or `clawbooks batch`.
2. **Read `policy.md` before classifying.** Category names, recognition rules, and thresholds come from policy.
3. **Provenance on every imported event.** `source_doc`, `source_row`, `source_hash` are mandatory for imports.
4. **Import every row.** Do not skip rows unless they are genuinely not financial events (headers, totals, blank lines). Reconcile your count against the source.
5. **Verify after every import.** `clawbooks verify` catches duplicates, sign errors, and balance mismatches.
6. **Confidence is honest.** Use `clear` only when classification is unambiguous. Prefer `inferred` over guessing. Use `unclear` when you truly can't classify.
7. **Source convention.** Use `statement_import` for bank/card statement imports unless the evidence clearly supports a more specific source.
